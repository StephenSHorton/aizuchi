# AIZ-57 spike — iOS audio capture plugin

**Status:** skeleton + design. Not device-verified.
**Branch:** `spike/aiz-57-ios-audio-plugin`
**Linear:** [AIZ-57](https://linear.app/aizuchi/issue/AIZ-57)

## TL;DR

- We **must** write a custom Tauri v2 mobile plugin. Every off-the-shelf alternative I evaluated (`tauri-plugin-audio-recorder`, `tauri-plugin-stt`, `tauri-plugin-mic-recorder`, `cpal`) fails on at least one of: streaming PCM, background-audio survival, or iOS lifecycle handling.
- **AVAudioSession mode:** `.measurement` with category `.playAndRecord` and options `[.mixWithOthers, .allowBluetooth, .defaultToSpeaker]`. `.measurement` is the only mode Apple documents as disabling the input signal-processing chain (AGC, noise suppression, EQ) — we want raw PCM going to whisper.cpp, not iOS-coloured audio. `.voiceChat` is wrong (it turns on AEC tuned for two-way calls; we have no playback to cancel). `.spokenAudio` is an output-side mode (audiobooks, podcast playback) and doesn't apply to recording.
- **Frame delivery:** use `tauri::ipc::Channel<AudioEvent>` rather than plugin events. Tauri's own docs say "Channels are designed to be fast and deliver ordered data ... used internally for streaming operations such as download progress, child process output and WebSocket messages" and that "The event system is not designed for low latency or high throughput situations." Source: <https://v2.tauri.app/develop/calling-frontend/>. At our rate (16 KB/s of PCM, ~10 frames/s) we're well inside the channel's design envelope; events would force a JSON-string-everything broadcast on every frame.
- **VAD placement:** in **Rust**, not Swift. Two reasons. (1) The bridge crossing is cheap (channel, not event); the I/O saving from gating on the Swift side is small. (2) Aizuchi's existing audio path in `src-tauri/src/audio.rs` already has a windowed silence policy. Putting VAD in Rust keeps a single source of truth for "what is silence" between desktop and mobile.
- **Top open question:** does iOS actually deliver continuous tap callbacks for a 30-min screen-locked session under `.measurement` + `UIBackgroundModes: [audio]`, or does it throttle? This is the one thing I can't validate without a physical device.

## 1. Why no off-the-shelf plugin works

| Plugin | Architecture | Why it doesn't fit |
|---|---|---|
| **`tauri-plugin-audio-recorder`** (brenogonzaga) | Desktop: WAV file via cpal. Mobile (iOS): `AVAudioRecorder` writing AAC/M4A to disk. No real-time PCM hook. | File-based by design. We need streaming frames in <100 ms windows for live transcription. |
| **`tauri-plugin-stt`** (brenogonzaga) | Wraps `SFSpeechRecognizer` (iOS) / Web Speech on web. | We don't want SFSpeechRecognizer — network-dependent for most languages, Apple-controlled, produces text not PCM. We want raw audio for our own whisper.cpp pipeline. |
| **`tauri-plugin-mic-recorder`** (ayangweb) | Desktop-only (cpal). No iOS module. | Not a candidate for iOS. |
| **`cpal`** directly | Cross-platform low-level audio I/O. | Works on iOS but thinly tested; we'd be reimplementing the AVFoundation lifecycle in Rust on top of `cpal` — strictly more work than owning `AVAudioEngine` in Swift. |

## 2. Plugin structure

The crate mirrors the official Tauri plugin layout, modelled on [`tauri-apps/plugins-workspace/plugins/haptics`](https://github.com/tauri-apps/plugins-workspace/tree/v2/plugins/haptics):

```
crates/tauri-plugin-aizuchi-audio/
├── Cargo.toml                       # tauri = "2", tauri-plugin = "2" build dep
├── build.rs                         # tauri_plugin::Builder, ios_path("ios")
├── permissions/
│   ├── default.toml                 # default permission set, hand-authored
│   └── autogenerated/.gitkeep       # build script writes allow-*/deny-* TOMLs here
├── src/
│   ├── lib.rs                       # init() / AizuchiAudioExt trait
│   ├── desktop.rs                   # PhantomData stub (no-op on macOS/Win/Linux)
│   ├── mobile.rs                    # ios_plugin_binding! + run_mobile_plugin dispatcher
│   ├── commands.rs                  # #[tauri::command] shims
│   ├── error.rs                     # Error enum, Serialize
│   └── models.rs                    # StartCaptureOptions, AudioEvent, etc.
└── ios/
    ├── Package.swift                # SPM manifest, depends on local Tauri package
    └── Sources/
        └── AizuchiAudioPlugin.swift # AVAudioSession + AVAudioEngine, ~330 lines
```

### `Cargo.toml`

The `tauri-plugin` build dep, the `links = "tauri-plugin-aizuchi-audio"` line (required so iOS static-library symbols don't collide with other plugins), and the iOS-only `tauri/wry` feature gate all match the official template.

### `build.rs`

```rust
const COMMANDS: &[&str] = &[
    "start_capture", "stop_capture",
    "check_permissions", "request_permissions",
];

fn main() {
    let result = tauri_plugin::Builder::new(COMMANDS)
        .ios_path("ios")
        .try_build();
    if !(cfg!(docsrs) && std::env::var("TARGET").unwrap_or_default().contains("ios")) {
        result.unwrap();
    }
}
```

### Rust side: `mobile.rs` dispatch pattern

Exactly the haptics pattern: `tauri::ios_plugin_binding!(init_plugin_aizuchi_audio)`, an `AizuchiAudio<R>(PluginHandle<R>)` newtype, each method forwards via `self.0.run_mobile_plugin("startCapture", payload)`.

The one wrinkle is **passing `Channel<AudioEvent>` *down* into Swift**. The payload type embeds the channel:

```rust
#[derive(Serialize)]
struct StartCapturePayload {
    #[serde(flatten)]
    options: StartCaptureOptions,
    on_event: tauri::ipc::Channel<AudioEvent>,
}
```

Tauri serializes a `Channel<T>` to a token the iOS plugin layer resolves back to a closure the Swift side can call. From Swift it's declared as `let onEvent: Channel` in the `Decodable` args class.

### Swift side: `AizuchiAudioPlugin.swift`

Subclasses `Plugin` from the `Tauri` package. Lifecycle:

- `load(webview:)` — registers `NotificationCenter` observers for interruption, route change, and media-services reset. Does **not** activate the session yet.
- `checkPermissions(_:)` / `requestPermissions(_:)` — uses the iOS 17 `AVAudioApplication.requestRecordPermission` API with a fallback to the iOS 15+ `AVAudioSession.recordPermission`.
- `startCapture(_:)` — configures `AVAudioSession`, installs the tap, starts the engine, returns a `{ id }`.
- `stopCapture(_:)` — tears down tap, stops engine, deactivates session.

The audio config:

```swift
try session.setCategory(
    .playAndRecord,
    mode: .measurement,
    options: [.mixWithOthers, .allowBluetooth, .defaultToSpeaker]
)
try session.setPreferredSampleRate(16_000)
try session.setPreferredIOBufferDuration(0.02)
try session.setActive(true, options: [.notifyOthersOnDeactivation])
```

Tap installation uses an explicit `AVAudioConverter` so we control sample-rate conversion (input node delivers hardware-native rate, typically 44.1/48 kHz):

```swift
input.installTap(onBus: 0, bufferSize: tapBufferSize, format: inputFormat) {
    [weak self] buffer, _ in
    self?.handleTap(buffer: buffer, converter: converter, outputFormat: outputFormat)
}
```

## 3. Frame delivery: events vs `Channel<T>`

### Numbers

At 16 kHz mono Int16: **32 KB/s** of PCM bytes (16,000 × 2). At 100 ms frame size, that's **10 frames/s of ~3.2 KB each**. Multiplying by JSON-array overhead (Int16 samples as decimal numbers ≈ 5x bloat), we're at ~160 KB/s on the wire — comfortably under what Tauri's Channel is designed for.

### Why Channel, not events

From Tauri v2 calling-frontend docs (<https://v2.tauri.app/develop/calling-frontend/>):

> Channels are designed to be fast and deliver ordered data. They are used internally for streaming operations such as download progress, child process output and WebSocket messages.
>
> The event system is not designed for low latency or high throughput situations.

Events also broadcast to *every* subscriber — a second window opening would silently start receiving raw PCM. Channel is point-to-point and ordered, which is what we want.

### Shared ring buffer?

Tempting (zero-copy) but huge complexity on the Swift/Rust seam: shared memory region + signalling primitive. Channel is good enough at this rate. Revisit if we ever go to 48 kHz stereo (~384 KB/s).

**Recommendation: `Channel<AudioEvent>`.** Skip shared memory.

### Binary payloads (Phase 2)

`AudioEvent::Frame` currently serializes `samples: Vec<i16>` to JSON — wasteful. Tauri's `Channel` supports `InvokeResponseBody::Raw` for binary; the optimization is to switch to `Channel<Vec<u8>>` with a fixed metadata header. JSON path is fine for the spike.

## 4. Interruption + route change handling

### `AVAudioSession.interruptionNotification`

Triggered by phone call, Siri, alarm, or another app taking over. The `.shouldResume` flag is the key signal: when iOS sets it, we restart the engine. When it's absent, we surface the interruption to Rust and wait for the user to manually re-start.

### `AVAudioSession.routeChangeNotification`

The non-obvious case is `.oldDeviceUnavailable` — user yanks AirPods out of their ears. iOS pauses the engine. We must:
1. Stop the (broken) engine.
2. Remove the tap (bound to a stale input format).
3. Reinstall the tap with the new default input (built-in mic).
4. Restart the engine.

Other reasons (`.newDeviceAvailable`, `.categoryChange`, etc.) are surfaced to Rust but don't force restart.

### `AVAudioSession.mediaServicesWereResetNotification`

Rare but catastrophic — `mediaserverd` died. The entire engine and session are invalid. Skeleton bails out and tells Rust to restart. **Full rebuild is a TODO.**

### App backgrounding

Handled implicitly by `UIBackgroundModes: [audio]`. Once that key is in `Info.plist` and the session is active when the screen locks, iOS keeps the tap callbacks coming and shows the red mic indicator. We don't subscribe to `UIApplication.didEnterBackgroundNotification` — nothing useful to do there.

## 5. VAD placement

**Decision: VAD lives in Rust.**

| Where | Pro | Con |
|---|---|---|
| **Swift (silence gate before bridge)** | Saves bridge traffic on silence. | Different code path from desktop; harder to tune; re-ship native module to change thresholds. |
| **Rust (gate after bridge)** | Single source of truth across desktop and mobile. Tunable from config. | All PCM crosses the bridge even during silence — ~16 KB/s wasted. |

The bridge-traffic concern is noise at our rate. Aizuchi's desktop path already does windowed silence policy in Rust (`src-tauri/src/audio.rs`); same place keeps semantics aligned.

The plugin **does ship** an opt-in `silenceGate: true` flag for Swift-side gating in case battery cost from streaming silence matters in practice. Default off.

## 6. Tauri capability config

In the consuming app's `src-tauri/capabilities/default.json`, add `"aizuchi-audio:default"` to `permissions`. The plugin's `permissions/default.toml` aggregates four per-command allow tokens. The autogenerated TOMLs are written into `permissions/autogenerated/` by the `tauri-plugin` build script.

## 7. `Info.plist` changes

iOS Tauri apps generate their `Info.plist` under `src-tauri/gen/apple/<app>.xcodeproj` after the first `tauri ios init` run (the `gen/` directory does **not** exist yet in this repo — `src-tauri/gen/` currently holds only `schemas/`). The pre-generation template lives at `src-tauri/Info.plist` and already has `NSMicrophoneUsageDescription` for the desktop path.

For iOS, the host app must additionally declare `UIBackgroundModes` with the `audio` string. Full snippet at [`./info-plist.xml`](./info-plist.xml).

## 10. Risks and open questions

1. **Background-audio continuity (THE big one).** Does iOS keep delivering tap callbacks for 30 min screen-locked under `.measurement` + `UIBackgroundModes: [audio]`? Apple has been tightening background-mode policies year over year. Needs device test.
2. **Tap callback thread + concurrency.** Tap callbacks fire on a real-time audio thread. We're capturing `[weak self]` and calling into the Tauri channel from that thread. Tauri Swift bridge thread-safety under that pattern is **not documented**. If we hit a race, the fix is to enqueue from the tap to a serial `DispatchQueue` before crossing the bridge.
3. **AVAudioConverter latency.** Adds a frame or two of buffering. Dominated by transcription model latency anyway, but worth measuring.
4. **`AVAudioApplication` vs `AVAudioSession` permission API on iOS 17+.** Newer API required for iOS 17; older one still works as deprecated shim. Code handles both via `#available`.
5. **Red mic indicator appears at session-activation, not tap-install.** So we should not activate the session until the user has tapped "start meeting".
6. **`AVAudioEngine` after `mediaServicesWereReset`.** Skeleton tears down and surfaces an error. Full recovery (rebuild engine, reactivate session, reinstall tap) is a substantial piece of code needing its own design pass.
7. **iOS Rust toolchain not installed in dev env.** Before next round, run `rustup target add aarch64-apple-ios x86_64-apple-ios aarch64-apple-ios-sim` and add `cargo check --target aarch64-apple-ios` to CI.
8. **AGPL compliance for the iOS App Store.** Aizuchi is AGPL-3.0-or-later. AGPL apps have historically been pulled (the VLC saga). Project-level decision; blocks shipping, not the spike.
9. **WWDC25 iOS 26 `bluetoothHighQualityRecording` option.** Worth adding to category options when we drop iOS 15 support; gives higher-quality AirPods input.
